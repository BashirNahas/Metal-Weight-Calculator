1.Technology stack

Nahas MetalHub is a single Flutter codebase compiled to a Progressive Web App (PWA) and, optionally, native Android/iOS apps from the same source.

The app has no backend server of its own — it is a static site. All "live" behaviour (market prices) is achieved by a scheduled GitHub Action that writes a JSON file into the same GitHub Pages branch, which the web app then fetches same-origin (no server, no database).

2.App architecture

The codebase follows a clean-architecture-style layering under lib/:

lib/
  core/          constants, theme colors, extensions, formatters
  models/        Metal, Shape, Calculation, MetalPrice, MeasurementUnit, PriceUnit
  services/      StorageService (SharedPreferences), MetalPriceService (HTTP)
  providers/     ChangeNotifier providers (Theme, Locale, Calculator, History,
                 Units, PriceUnit, MarketPrices)
  screens/       Calculator, Market Prices, History, Settings, Splash
  widgets/       Reusable UI: selectors, cards, dialogs
  l10n/          Hand-rolled Arabic/English localization (AppLocalizations)

State flows one-directionally: a screen reads a provider via context.watch<X>(), the user triggers an action, the provider mutates its state and calls notifyListeners(), and persists the change to SharedPreferences through StorageService. The app defaults to Arabic with a full English translation toggle in Settings, and supports light/dark/system theme.

4.Shapes & formulas

All formulas compute a volume in cm³ first; weight is then volume × density ÷ 1000 (the ÷1000 converts grams to kilograms, since density is g/cm³).

ShapeInputsVolume formula
Rectangular plateLength, Width, ThicknessL × W × T
Circular discDiameter, Thicknessπ × r² × T, r = D/2
Square barSide, LengthS² × L
Round barDiameter, Lengthπ × r² × L
Hexagonal barAcross-flats, Length(√3/2) × AF² × L
Round pipeOuter ∅, Wall thickness, Lengthπ/4 × (OD² − ID²) × L, ID = OD − 2×wall
Square tube newOuter side, Wall thickness, Length(S² − (S−2×wall)²) × L
Hollow shapes (pipe / square tube) return a weight of 0 if the wall thickness is large enough that the inner cavity would have zero or negative size (i.e. wall × 2 ≥ outer dimension) — this guards against a physically impossible input rather than producing a negative weight.

Implementation: lib/models/calculation.dart, function Calculation.compute().

5.How a calculation works end-to-end

1. Pick metal
→
2. Pick shape
→
3. Enter dimensions
→
4. Tap Calculate
→
5. Result + history
  1. The calculator screen renders only the input fields relevant to the selected shape (e.g. a round bar only asks for diameter + length, never thickness).
  2. Every dimension field is entered in the user's chosen measurement unit (mm / cm / inch, see §6). Before computing, CalculatorProvider.calculate() converts every value to centimeters using a fixed factor (mm ×0.1, cm ×1, inch ×2.54), because all density figures and formulas are defined in cm/g internally.
  3. Calculation.compute() runs the matching geometric formula from §4 to get a volume in cm³, then multiplies by the metal's density and divides by 1000 to get kilograms.
  4. The result is shown in a dialog (copy / share / favorite) and automatically saved to a local on-device history (capped at the last 100 entries), so nothing is sent to any server for this step — it is 100% on-device math.

6.Measurement & price display units

Measurement unit (Settings → Calculator)

Controls what unit the dimension input fields (length, width, diameter, thickness...) are interpreted in. Choices: mm, cm (default), inch. This only changes how typed numbers are interpreted — the underlying formulas always run in centimeters internally, so switching the setting does not change a previously saved calculation, only how new input is read.

Price display unit (Settings → Prices)

Controls which unit the Market Prices screen shows as the large/primary figure per metal: Tonne (default — matches the original app behaviour), Kilogram, or Ounce (troy ounce, standard for precious metals). The secondary (smaller) figure automatically shows a complementary unit for quick cross-reference. Conversions:

price_per_kg    = price_per_tonne / 1000
price_per_ounce = price_per_tonne / 32150.7466   // troy ounces per metric tonne

7.Live prices & the metals.dev API

Market prices are sourced from metals.dev, a third-party commodities pricing API.

EndpointGET https://api.metals.dev/v1/latest
Query paramsapi_key, currency=USD, unit=tonne
AuthAPI key as a query parameter (no OAuth/headers)
API key (masked)GSXS•••••••••••••••••••••I2G0TA redacted
Rate limit / plan tierNot independently verifiable from this document — see the callout below
Why the key above is masked: this is a publicly shareable documentation file. Embedding the real key in plaintext here would leak the live production secret to anyone who opens this file. The real key only ever lives in two places: the METALS_API_KEY GitHub Actions secret, and as a --dart-define build-time value (see §9) — never committed to source control in plaintext.
Rate limits / pricing tier: metals.dev's public pricing and rate-limit pages returned an HTTP 403 when checked programmatically while writing this document, so exact numbers can't be confirmed here. Check your actual usage and limits directly in your metals.dev account dashboard. The app's own polling is intentionally conservative — see the pipeline section below — specifically to stay well under typical free/low-tier API limits.

Response shape (simplified)

{
  "timestamps": { "metal": "2026-06-29T12:00:00Z" },
  "metals": {
    "copper": 9821.4,
    "gold": 3981.2,     // USD per troy-oz (NOT per tonne — see below)
    "silver": 39.8,
    ...
  }
}

One quirk the app accounts for: metals.dev always returns gold, silver, platinum and palladium in USD per troy-ounce, even though the request asked for unit=tonne. MetalPriceService detects these four keys and multiplies them by 32,150.7466 (troy ounces per metric tonne) to normalize everything to USD/tonne before the app does anything else with the numbers.

8.Price automation pipeline

Because this is a static site with no backend, "live" prices are produced by a scheduled GitHub Action rather than an API call made directly from every visitor's browser (which would expose the API key and burn through the rate limit per-visitor).

GitHub Actions cron (every 4h)
→
calls metals.dev with the secret key
→
writes prices.json
→
commits to gh-pages
→
app fetches it same-origin

Workflow file: .github/workflows/update-prices.yml. Schedule: 0 */4 * * * (every 4 hours), plus a manual workflow_dispatch trigger.

In the Flutter app, MetalPriceService._fetchWeb() tries, in order:

  1. Same-origin prices.json — the file the GitHub Action just produced. No CORS issue since it's served from the same domain as the app itself. This is the normal path.
  2. CORS proxies (allorigins.win, corsproxy.io) as a fallback that calls metals.dev directly from the browser, only used if step 1 fails (e.g. file missing or a network hiccup).

The app also caches the last successfully fetched prices locally (StorageService, 15-minute cache window) so the Market Prices screen has something to show instantly on reopen, with a background refresh.

Rebuilding & redeploying after a code change

flutter build web --base-href /Metal-Weight-Calculator/ --release \
  --dart-define=METALS_API_KEY=<the real key, never committed>

# then copy build/web/* into the gh-pages branch and push

The --dart-define flag injects the API key at build time only — it ends up compiled into the JS bundle (necessary for the browser to call the API directly as a fallback path), but it is never written to any tracked source file.

10.Security & secrets handling

Where the real API key lives:
  • As the GitHub Actions repository secret METALS_API_KEY, used by update-prices.yml.
  • Passed at local build time via --dart-define=METALS_API_KEY=..., never typed directly into a source file.
Files that must stay out of version control (already in .gitignore): .env, key.properties, secrets.dart, android/key.properties.

AppConstants.metalsApiKey is declared as String.fromEnvironment('METALS_API_KEY', defaultValue: '...') — the defaultValue exists purely so the app still builds and runs for local development without needing the flag every time; production deploys always override it explicitly via --dart-define.
